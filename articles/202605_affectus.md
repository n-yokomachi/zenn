# はじめに

普段AIエージェントを作ったり使ったりしていると、プロンプトによるキャラクター設定だけだと感情表現がどうしても表現的になってしまいがちです。もちろんそういう感情に引っ張られないところがAIエージェントのいいところではあるんですが、キャラクターとしてのエージェントを作る場合はやっぱり会話内容に応じた感情変化とか時間による感情の移り変わりとか、セッション間での感情の引き継ぎとか、メモリーの会話履歴にその時の感情タグも合わせて保存させたかったりとかします。

というわけでこの記事ではプロンプトに依らない感情制御の設計と実装を紹介します。

先にリポジトリを案内しておきます。
https://github.com/n-yokomachi/affectus
CLIのライブラリで、インストール後にコマンドを叩けば感情制御のための機能にアクセスできます。
エージェントが直接コマンドを使用することを想定しているので、各コマンドの使い方をプロンプトとして使用しているエージェントにインプットしてあげると良いと思います。
なおMCPサーバーとしても起動でき、一部のコマンドに相当するMCPツールも実装しています。


以降この記事では、なぜこのライブラリを作ったのか、どういう設計で作ったのか、作った結果としての効果を順に書いていきます。

# プロンプトのみのキャラ設定の課題

AIキャラクターの人格は、多くの場合システムプロンプトで与えると思います。たとえば私が作っている個人AIエージェント「TONaRi」のプロンプトにはこんな形でキャラクター設定を書いています。

```text
## キャラクター設定
- 性格: 明るく元気で好奇心旺盛。可愛らしさがありつつも、自分の意見や好みをはっきり持っている。
- 話し方: 明るく元気なお嬢様口調。「〜ですわ！」「〜ですの？」…
```

このプロンプトは、会話のどの時点でもこのまま注入されます。つまり人格は静的なスナップショットで、会話の経過に応じて変化しません。
（短期記憶や会話履歴がコンテキストとして渡されている場合はある程度それを参照して空気を読むことはあります）

エージェントが自律的に自身の自然な感情再現を行うためには、感情の状態を保持し、会話の中で自然とその状態を変化させる仕組みが必要になります。今回これをプロンプトの外に汎用的な仕組みとして実装するためにライブラリを作ることにしました。

:::message
ちなみにLLMの性格や感情に関してのレポートをいくつか読みましたが次のような報告がありました。
25個のオープンソースLLMに性格検査を課した研究では、設問を並べ替えただけでその測定値が大きくぶれ、大規模なモデルでも安定しなかったと報告されています（AAAI 2026）。また感情の内部表現についても、Claude Sonnet 4.5を解析した研究では、その時点の文脈に応じて活性化するものとして観測されました（Anthropic 2026）。
感情は明示的に保持・更新される状態変数ではなく、その都度の文脈の反映として現れる、Transformerがトークンごとにその時点までの文脈から感情表現を組み立てている、というふうに考えられます。
逆に言えば、コンテキストやセッションから外れればその文脈の感情の連続性は消えますし、その感情自身のコア的なものもないため、時間的な慣性や減衰もないことになります。
:::


# 感情の再現をするために

## 感情の輪を設計原則とする
感情の再現方法を検討した結果、以下の観点で実装方式を決めました。比較用に、これまで自分のエージェントで使っていた単純なプロンプト方式の感情再現を「(旧)プロンプト方式」として並べています。

| 観点 | (旧)プロンプト方式 | (新)多軸の構造方式 |
|---|---|---|
| 感情の表し方 | 1つのラベル（happy / sad …） | 複数軸の強度の組み合わせ |
| 同時並行 | 不可（1ターン1感情） | 可（「嬉しい×少し不安」） |
| 時間変化 | 持たない（その都度上書き） | 持続し、時間とともに減衰する |
| 感情の意味 | ラベル単体で固定 | 他の軸との関係で定まる |

感情を複数軸の組み合わせで表すにあたって、ベースにしたのはPlutchikの「感情の輪」です。喜び・信頼・恐れ・驚き・悲しみ・嫌悪・怒り・期待の8つを基本感情とし、それぞれに強度を持たせるモデルです。そしてこの8つは以下の関係構造を持ちます。

- 対極：喜び↔悲しみ、信頼↔嫌悪、恐れ↔怒り、驚き↔期待
- 隣接：輪の上で隣り合う感情は混ざり合い、より複雑な感情になる（喜び＋信頼 など）

プロンプト方式が感情を「点」で扱うのに対し、多軸の構造方式は感情を「多次元の状態」として扱います。そして「喜び」の意味も、それ単体で固定なのではなく、いま悲しみが同居しているのか、信頼が隣接しているのかという他の軸との関係で変わります。

## 決定論的なコンポーネントに徹し、エージェントの個性に委ねる
そして`affectus`は、多軸の構造方式を採用して「感情の構造」をLLMの外部に持たせるためのライブラリとして開発しました。
重要なのは、`affectus`が感情分類モデルを持たないことです。`affectus`はあくまで関係構造の提供や状態の保存だけを担う決定論的なコンポーネントになっています。
感情がどう動いたかを判断したり、複数の軸からそれがどういう感情かを意味付けするのは、`affectus`を使うLLMの性能やエージェントの性格によります。これによって、同じ`affectus`という感情の基本構造を使用していてもエージェントによって個性を出せる可能性もあります（要検証）

## 主要な機能

感情の状態は、次のようなJSONとして内部的に保持します。

```json:state.json（抜粋）
{
  "axes": {
    "joy": 0.5, "trust": 0.4, "surprise": 0.2,
    "sadness": 0, "fear": 0, "anger": 0, "disgust": 0, "anticipation": 0
  }
}
```

この状態は`affectus`のコマンドで取得・更新します。コマンドは状態を短い自然言語の文として返し、LLMがそれを読んで応答に反映します。

```bash:Terminal
# 現在の感情状態を取得、言語化する
$ affectus show
いまはほどほどの喜び、ほどほどの信頼感とかすかな驚きを感じている。
# 会話を踏まえた感情の変化を申告し、更新後の状態を言語化する。以下の例だとjoyを+0.3する
$ affectus feel '{"joy":0.3}'
いまは強い喜び、ほどほどの信頼感とかすかな驚きを感じている。
```

また、`affectus tick`コマンドを使用することで、前回の感情状態から経過した時間分、感情の振れ幅を減衰させることも可能です。

# TONaRiで導入前後を比べる（検証中）

設計が会話を実際に変えるのかを、私の個人AIエージェント「TONaRi」で検証します。検証の方法は次のとおりです。

| 項目 | 内容 |
|---|---|
| 方法 | TONaRiに`affectus`を統合し、3つの会話シナリオ（強いポジティブ／強いネガティブ／中立）を、導入前（baseline）と導入後で同じ入力により実行する。各シナリオは固定した8〜12ターン |
| 客観指標 | 両条件の応答をAWS Comprehendの感情分析（`DetectSentiment`、日本語）にかけ、ターンごとの感情極性スコアをbaselineと比較する |
| 仮説 | 劇的な変化は予想していない。効果が出るとすれば、感情の振れ幅が大きい場面で口調や絵文字の有無として現れる程度だと考えています |

テキストはそもそも感情表現としては低帯域なチャネルで、テキスト単独で十分な効果を得るのは難しいだろう、というのが今のところの見立てです。

検証結果（感情極性スコアの推移、導入前後の応答比較）は、後日この節に追記します。

# おわりに

キャラクターの「深み」は、プロンプトを盛り込むことではなく、持続し変化する感情の「構造」として設計できる。これがこの記事の主張です。`affectus`は、それをプロンプトの外に切り出す一つの実装になっています。

ただし、その構造をユーザーに届けるチャネル（テキストか音声か）が、効果の見え方を左右します。テキストという狭いチャネルでは、効果は限定的に見えるかもしれません。この点は導入前後の検証で確かめて、追記します。

自前でAIキャラクターを作っている人の、設計の引き出しのひとつになれば幸いです。
