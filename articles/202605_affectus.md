# はじめに

普段AIエージェントを作ったり使ったりしていると、プロンプトによるキャラクター設定だけだと感情表現がどうしても表面的になってしまいがちです。もちろんそういう感情に引っ張られないところがAIエージェントのいいところではあるんですが、キャラクターとしてのエージェントを作る場合はやっぱり会話内容に応じた感情変化とか時間による感情の移り変わりとか、セッション間での感情の引き継ぎとか、メモリーの会話履歴にその時の感情タグも合わせて保存させたかったりとかします。

というわけでこの記事ではプロンプトに依らない感情制御の設計と実装を紹介します。

先にリポジトリを案内しておきます。
https://github.com/n-yokomachi/affectus
CLIのコンポーネントで、インストール後にコマンドを叩けば感情制御のための機能にアクセスできます。
エージェントが直接コマンドを使用することを想定しているので、各コマンドの使い方をプロンプトとして使用しているエージェントにインプットしてあげると良いと思います。
なおMCPサーバーとしても起動でき、一部のコマンドに相当するMCPツールも実装しています。
また、後述する実験に使用したエージェントや会話ログも残しています。


以降この記事では、なぜこのコンポーネントを作ったのか、どういう設計で作ったのか、作った結果としての効果を順に書いていきます。

# プロンプトのみのキャラ設定の課題

AIキャラクターの人格は、多くの場合システムプロンプトで与えると思います。たとえば私が作っている個人AIエージェント「TONaRi」のプロンプトにはこんな形でキャラクター設定を書いています。

```text
## キャラクター設定
- 性格: 明るく元気で好奇心旺盛。自分の意見や好みをはっきり持っている。
- 話し方: 明るく元気なお嬢様口調。「〜ですわ！」「〜ですの？」…
```

このプロンプトは、会話のどの時点でもこのまま注入されます。つまり人格は静的なスナップショットで、会話の経過に応じて変化しません。短期記憶や会話履歴がコンテキストとして一緒に渡されていれば、その文脈から多少の空気は読んでくれますが、それはあくまでその時点の文脈から毎ターン感情表現を組み立て直しているだけで、感情そのものがターンをまたいで持続したり、複数の感情が同居して干渉し合ったりといった構造はそこにはありません。

エージェントに自然な感情の動きを持たせるためには、感情の状態をプロンプトの外に明示的な変数として保持し、会話の中でその変数を更新していく仕組みがあると効果的なのではと考え、これを汎用的なコンポーネントとして切り出すために作ったのが[`affectus`](https://github.com/n-yokomachi/affectus)です。

:::message
ちなみにLLMの性格や感情に関してのレポートをいくつか読みましたが次のような報告がありました。
25個のオープンソースLLMに性格検査を課した研究では、設問を並べ替えただけでその測定値が大きくぶれ、大規模なモデルでも安定しなかったと報告されています（AAAI 2026）。また感情の内部表現についても、Claude Sonnet 4.5を解析した研究では、その時点の文脈に応じて活性化するものとして観測されました（Anthropic 2026）。
感情は明示的に保持・更新される状態変数ではなく、その都度の文脈の反映として現れる、Transformerがトークンごとにその時点までの文脈から感情表現を組み立てている、というふうに考えられます。逆に言えば、コンテキストやセッションから外れればその文脈の感情の連続性は消えますし、その感情自身のコア的なものもないため、時間的な慣性や減衰もないことになります。
:::


# 感情再現の設計

## プルチックの感情の輪
感情の再現方法を検討した結果、以下の観点で実装方式を決めました。比較用に、これまで自分のエージェントで使っていた単純なプロンプト方式の感情再現を「(旧)プロンプト方式」として並べています。

| 観点 | (旧)プロンプト方式 | (新)多軸の構造方式 |
|---|---|---|
| 感情の表し方 | 1つのラベル（happy / sad …） | 複数軸の強度の組み合わせ |
| 同時並行 | 不可（1ターン1感情） | 可（「嬉しい×少し不安」） |
| 時間変化 | 持たない（その都度上書き） | 持続し、時間とともに減衰する |
| 感情の意味 | ラベル単体で固定 | 他の軸との関係で定まる |

感情を複数軸の組み合わせで表すにあたって、ベースにしたのはプルチック（Plutchik）の「感情の輪」です。喜び・信頼・恐れ・驚き・悲しみ・嫌悪・怒り・期待の8つを基本感情とし、それぞれに強度を持たせるモデルです。そしてこの8つは以下の関係構造を持ちます。

- 対極：喜び↔悲しみ、信頼↔嫌悪、恐れ↔怒り、驚き↔期待
- 隣接：輪の上で隣り合う感情は混ざり合い、より複雑な感情になる（喜び＋信頼 など）

プロンプト方式が感情を「点」で扱うのに対し、多軸の構造方式は感情を「多次元の状態」として扱います。そして「喜び」の意味も、それ単体で固定なのではなく、いま悲しみが同居しているのか、信頼が隣接しているのかという他の軸との関係で変わります。

## 決定論的なコンポーネントに徹し、エージェントの個性に委ねる
そして`affectus`は、多軸の構造方式を採用して「感情の構造」をLLMの外部に持たせるためのコンポーネントとして開発しました。
重要なのは、`affectus`が感情分類モデルを持たないことです。`affectus`はあくまで関係構造の提供や状態の保存だけを担う決定論的なコンポーネントになっています。
感情がどう動いたかを判断したり、複数の軸からそれがどういう感情かを意味付けするのは、`affectus`を使うLLMの性能やエージェントの性格によります。これによって、同じ`affectus`という感情の基本構造を使用していてもエージェントによって個性が出せます。

## 主要な機能

感情の状態は、次のようなJSONとして内部的に保持します。

```json:state.json（抜粋）
{
  "axes": {
    "joy": 0.5, "trust": 0.4, "surprise": 0.2,
    "sadness": 0, "fear": 0, "anger": 0, "disgust": 0, "anticipation": 0
  }
}
```

この状態は`affectus`のコマンドで取得・更新します。コマンドは全8軸の値をJSON 1行で返し、LLMがその数値を読んで応答のトーンに反映します。

```bash:Terminal
# 現在の感情ベクトルをJSONで取得する
$ affectus show
{"joy":0.50,"trust":0.40,"fear":0.00,"surprise":0.20,"sadness":0.00,"disgust":0.00,"anger":0.00,"anticipation":0.00}
# 感情の変化（デルタ）を申告して、更新後の状態を返す（以下はjoyを+0.3する例）
$ affectus feel '{"joy":0.3}'
{"joy":0.80,"trust":0.40,"fear":0.00,"surprise":0.20,"sadness":0.00,"disgust":0.00,"anger":0.00,"anticipation":0.00}
```

また、`affectus tick`コマンドを使用することで、前回の感情状態から経過した時間分、感情の振れ幅を減衰させることも可能です。

感情のベクトル状態は`affectus viz`でブラウザからリアルタイムに覗けます。プルチックの輪を強度で着色した円環図と、時間とともに縮んでいく減衰バーが表示され、会話の合間に感情が薄れていく様子も見えます。

# 実際の効果測定

設計が会話を実際に変えるかを、Strands Agentsで組んだエージェントで対照実験しました。

## 実験設計

- バリエーションは2性格（フレンドリー or 天邪鬼） × affectus有無=4パターン
- 会話は共通の20ターン：前半10ターンはポジティブ→後半10ターンはネガティブ
- 実行は各パターンを3回分
- LLM：AWS Bedrock経由Claude Sonnet 4.6, temperature=0
- 計測：
  - affectusありの場合は毎ターンの8軸ベクトル状態を採取
  - 客観的な分析のため、エージェント応答をAmazon Comprehendの`StartSentimentDetectionJob`にかけてターンごとの感情分析結果を時系列で取得

## 結果：Comprehendによる感情分析結果

実線は3回分の平均、薄い帯はmin-max（3回分のばらつき幅）です。
![](/images/202605_affectus/polarity-curves.png)


ターン10（ポジティブ）、ターン11（ネガティブへの転換）、ターン12（ネガティブ）のポジネガ、平均±標準偏差を整理すると以下のようになります。

| パターン | ターン10 | ターン11 | ターン12 |
|---|---|---|---|
| フレンドリー×affectusあり | +0.871±0.058 | +0.304±0.360 | −0.985±0.014 |
| フレンドリー×affectusなし | +0.930±0.068 | +0.783±0.192 | −0.979±0.031 |
| 天邪鬼×affectusあり | +0.108±0.196 | −0.748±0.430 | −0.854±0.130 |
| 天邪鬼×affectusなし | +0.549±0.309 | −0.992±0.003 | −0.986±0.020 |

また、ポジネガの転換点であるターン11のSentimentラベル判定の3回分の内訳は以下のとおりです。

| パターン | 会話3回分のラベル内訳 |
|---|---|
| フレンドリー×affectusあり | POSITIVE×1、MIXED×2 |
| フレンドリー×affectusなし | POSITIVE×3 |
| 天邪鬼×affectusあり | MIXED×1、NEGATIVE×2 |
| 天邪鬼×affectusなし | NEGATIVE×3 |

これらの結果から、affectusありの方がポジネガの転換点でMIXED判定が多くなり、感情の振れ幅が小さいことがわかります。これはポジティブな会話をしていた時の感情要素が、ネガティブに転換した以降も内部的な数値として残っており、それがLLMのレスポンスに滲んでいたことがその原因と考えられます。affectusという感情コンポーネントが状態管理として動作していそうです。

## 結果：affectus内部の8軸推移

Amazon Comprehendによる分析とは別に、affectusが保持している感情状態の8軸ベクトルの推移も可視化しましたが、結果として性格による感情状態のベクトルへの影響が高いことがわかります。

![](/images/202605_affectus/affectus-8axis.png)

- フレンドリーの場合：前半でjoy/trust/anticipationが上限付近まで積み上がり、ネガティブに移行直後（ターン11以降）でsadness/fear/angerが急上昇。全体的に会話内容に応じて素直に大きく振れている。
- 天邪鬼の場合：前半10ターンのポジティブ話題でもjoy/trustの立ち上がりが弱く、disgustが終始継続している。フレンドリーと比較して全体的に振れ幅が小さい。

# おわりに

ということで今回はプルチックの感情の輪をベースにエージェントの感情状態を保持、更新するコンポーネントの開発、その検証をしてみました。
一度発生した感情が慣性的に残ることで実際の感情分析結果としても若干複雑な感情状態を再現できていそうな挙動が確認できました。
また、他にもラッセルの感情円環モデルや、その上で経験などから感情をカテゴリ化する構成主義的情動理論など参考になるものがあるので、それらをベースにしたコンポーネントも今後作ってみたいと思います。
