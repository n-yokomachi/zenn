# はじめに

現在進行形でTONaRiというパーソナルAIエージェントを個人開発しています。Strands Agents + Amazon Bedrock AgentCoreで構築しており、フロントエンドにはAITuberKitを使用したVRM制御を行っています。
![](https://storage.googleapis.com/zenn-user-upload/6d6a131e15f1-20260307.png)

パーソナルAIエージェントとして日常的な作業をサポートしてもらうため、様々なツールを作っては直接繋げていたのですが、気づけばエージェント呼び出し時の入力トークンが爆増していてコストがだいぶキツくなっていました。
![](https://storage.googleapis.com/zenn-user-upload/1fa4d12f457d-20260307.png)
*今は下がっているがその前の予測は$120くらいだった*

というわけでこの記事では、ツールの増加による入力トークン増加問題と、サブエージェントへの切り出しでなんとか抑えようとしている話をしたいと思います。


# アーキテクチャ概要

まずTONaRi（パーソナルAIエージェント）の構成ですがこんな感じです

```
Frontend (Next.js + VRM 3Dアバター)
  → Next.js API Route
    → AgentCore Runtime（Strands Agent）
      → AgentCore Gateway → Lambda関数（各種ツール）
      → AgentCore Memory (STM/LTM)
```

エージェント本体はBedrock AgentCore Runtimeにデプロイされたコンテナとして動作し、外部ツールはAgentCore Gatewayを経由してLambda関数として実装しています。
ツールの追加は、Lambda関数を書いてGatewayにターゲットとして登録するだけです。AgentCore Gatewayはこの手軽さが便利ですね。


# 繋がれた数多のツール

AgentCore Gatewayを使うと、Lambda関数をエージェントのツールとして公開できます。

```python
from strands.tools.mcp import MCPClient
from mcp_proxy_for_aws.client import aws_iam_streamablehttp_client

def create_mcp_client(gateway_url: str, region: str) -> MCPClient:
    def create_transport():
        return aws_iam_streamablehttp_client(
            endpoint=gateway_url,
            aws_region=region,
            aws_service="bedrock-agentcore",
        )
    return MCPClient(create_transport)
```

で、今回追加したツールが以下となります。

| ドメイン | ツール | 個数 |
|---------|--------|------|
| タスク管理 | 一覧取得、追加、完了、更新 | 4 |
| カレンダー | 予定一覧、空き確認、作成、更新、削除、日程提案 | 6 |
| Gmail | メール検索、取得、下書き作成、アーカイブ | 4 |
| Notion | ページ検索、取得、作成、更新、DB検索、DB取得 | 6 |
| Twitter | ツイート取得、投稿 | 2 |
| 日記 | 保存、取得 | 2 |
| 日付計算 | 現在日時取得、日付計算、日付範囲一覧 | 3 |
| Web検索 | Web検索 | 1 |
| 合計 | | 28 |

各ツールを単発で呼び出すことももちろん可能ですし、例えば「Webでレシピ検索してブックマークに保存して買い物メモ作って、タスクに買い物を追加して」という指示をすると、
1. Web検索ツールでレシピを検索
2. NotionのブックマークページにURLを保存
3. レシピの内容から買い物リストを作成してNotionのメモページに保存
4. 買い物タスクをTONaRi自身のタスクリストに追加

の一連の作業をやってくれます。

AIエージェントがツールの間に介在し、ユーザーからの雑なリクエストを解釈して各ツールの橋渡しをしてくれるというのが現在日常的にAIエージェントを使っていて一番便利に感じている点です。


# 入力トークンの爆増

しかし、便利になっていくAIエージェントの裏では着実にコストが積み重なっていました。
BedrockのAPIを呼び出す際、入力トークンを構成する要素は大きく分けて以下の4つです。

1. システムプロンプト: エージェントのキャラクター設定、行動ルール
2. ツール定義: 全ツールの名前、説明文、パラメータのJSONスキーマ
3. 長期記憶（LTM）: 過去の会話から抽出されたエピソードとか事実情報
4. 会話履歴（STM）: 現在のセッションの内容

一番のネックはツール定義のトークン量でした。
Claude Codeに計算させた結果、現時点で直接エージェントに接続している28ツールで約5,000トークンに達していました。


## 実際の内訳を見てみる

28個のツールを抱えたモノリシックなエージェントの1回あたりの入力トークン内訳を概算してみます。

| 項目 | 推定トークン数 |
|------|-------------|
| システムプロンプト（キャラ設定 + 全ドメインの行動ルール） | ~3,500 |
| ツール定義（28ツール × スキーマ） | ~5,000 |
| LTM検索結果 | ~1,500 |
| 会話履歴（10ターン分） | 可変（~5,000〜30,000） |

プロンプト、ツール、LTMはほぼ固定で毎回入力されるので、最大10,000トークンが毎メッセージで入力されている計算です。
例えば1日に約100ターンの会話をしたら呼び出されるので、月間の固定コストだけでも：

```
10,000 tokens × 100 turns/day × 30 days = 30,000,000 tokens/month
```

使用しているClaude Haiku 4.5のBedrock入力トークン単価（$1.10/1M tokens、日本クロスリージョン推論）で計算すると、固定コストだけで月$33です。個人開発で毎月約5,000円が、使わないかもしれないツール定義などに消えていることになります。


# サブエージェントに切り出す

とりあえずメインエージェントが直接参照するツール定義を減らすため、ドメインごとにサブエージェントを作成し、メインエージェントからは`@tool`でサブエージェントを呼び出すよう修正しました。

```
【Before: モノリシック】
メインエージェント
├── システムプロンプト（全ドメインのルール）
└── 28個のツール ← 毎回全部送信

【After: サブエージェント分割】
メインエージェント
├── システムプロンプト（汎用ルールのみ）
├── DateTool（3ツール）      ← 頻出なのでメインに残す
├── TavilySearch（1ツール）  ← 同上
├── task_agent      ← @toolとして定義（4ツール）
├── calendar_agent  ← @toolとして定義（6ツール）
├── gmail_agent     ← @toolとして定義（4ツール）
├── notion_agent    ← @toolとして定義（6ツール）
├── diary_agent     ← @toolとして定義（2ツール）
├── briefing_agent  ← @toolとして定義（中身は複数ドメインのツール）
└── twitter_agent   ← @toolとして定義（2ツール）
```


## サブエージェントの実装

Strands Agentsの`@tool`デコレータを使えば、サブエージェントをメインエージェントのツールとして定義できます。

```python
@tool
def calendar_agent(request: str) -> str:
    """Googleカレンダーのサブエージェント。予定の一覧取得、空き確認、作成、更新、削除を行う。

    Args:
        request: オーナーのカレンダーに関するリクエスト
    """
    try:
        agent = Agent(
            model=BedrockModel(
                model_id="jp.anthropic.claude-haiku-4-5-20251001-v1:0",
                region_name="ap-northeast-1",
                streaming=True,
            ),
            system_prompt="あなたはGoogleカレンダーの専門アシスタントです。...",
            tools=_calendar_tools,  # カレンダー系ツールのみ
            callback_handler=None,
        )
        result = agent(request)
        return str(result)
    except Exception as e:
        return f"カレンダー操作でエラーが発生しました: {e}"
```


## システムプロンプトの削減

ドメインごとにサブエージェントを切り出したため、メインエージェントのシステムプロンプトからサブエージェントにツール定義を移譲できました。これによりメインのシステムプロンプトを削減できています。

Before: メインプロンプトにすべてのドメインルールを記載
```
- カレンダー操作時のルール（重複確認、削除前確認、etc.）
- Gmail操作時のルール（下書きのみ、日付検索の注意点、etc.）
- Notion操作時のルール（プロパティ形式、データベースマッピング、etc.）
- ブリーフィング手順（5セクション分の詳細手順）
- 日記作成手順（ヒアリング → 生成 → 保存のフロー）
- ...
```

After: メインプロンプトにはサブエージェントの一覧と委任ルールだけ
```
## サブエージェント連携
- task_agent: タスク管理（一覧取得、追加、完了、更新）
- calendar_agent: Googleカレンダー（予定の取得、作成、更新、削除）
- gmail_agent: Gmail（メール検索、取得、下書き作成）
- ...

### 委任のルール
- サブエージェントへのリクエストは具体的に記述する
- サブエージェントの結果はそのまま伝えず、自分の言葉で言い換える
```

これでシステムプロンプトは約7,400文字 → 約3,800文字に削減できました。



## コスト削減効果

まず、メインエージェントの1回あたりの固定コストを比較します。

| 項目 | Before（モノリシック） | After（サブエージェント分割） |
|------|---------------------|--------------------------|
| システムプロンプト | ~3,500 tokens | ~2,000 tokens |
| ツール定義 | 28ツール（~5,000 tokens） | 12ツール（~2,500 tokens） |
| LTM検索結果 | ~1,500 tokens | ~1,500 tokens |
| 固定コスト合計 | ~10,000 tokens | ~6,000 tokens |

この4000トークンは純粋に削除されたわけではなく、サブエージェント側に移動した形となります。サブエージェント側の入力トークンはそれぞれ以下の通りです
:::message
この辺はClaude Codeで出した値をそのまま貼ってます
:::

| サブエージェント | プロンプト | ツール定義 | リクエストメッセージ | 合計 |
|----------------|-----------|-----------|-----------|---------|
| task_agent | ~400 | ~400 | ~100 | ~900 |
| calendar_agent | ~400 | ~850 | ~100 | ~1,350 |
| gmail_agent | ~400 | ~400 | ~100 | ~900 |
| notion_agent | ~400 | ~700 | ~100 | ~1,200 |
| briefing_agent | ~500 | ~2,500 | ~100 | ~3,100 |
| diary_agent | ~400 | ~200 | ~100 | ~700 |
| twitter_agent | ~400 | ~150 | ~100 | ~650 |


ということで、単純に合計するとAfterのほうが合計トークンが多くなるわけですが、今回の対応は毎回入力されるトークンを減らすことにあります。例えばブリーフィングコマンドではGmail, Googleカレンダー, タスクリストなどをいっぺんに参照したり、支出計算などをするためのルールがあって、定義の重いツールなのですが、Beforeではそれが毎回呼ばれていたものを、Afterでは必要な時だけ呼ばれるようになっています。
１日１回しか使わないツール定義を毎回の呼び出しで発生させないことによってコスト削減をしています。

## 月間コストへの影響

1日約100回の呼び出しで計算します。

```
【メインエージェントの固定コスト削減（毎回発生）】
  4,000 tokens/call × 100 calls/day × 30 days = 12,000,000 tokens/month

【サブエージェント呼び出しの追加コスト（呼ばれた時だけ発生）】
  呼び出しの約60%（60回/日）でサブエージェントが1回起動すると仮定
  平均 900 tokens/call × 60 calls/day × 30 days = 1,620,000 tokens/month
  ※briefing_agent（~3,100 tokens）は1日1回のみのため平均から除外し別計上
  briefing: 3,100 tokens × 30 days = 93,000 tokens/month

【差し引き】
  12,000,000 - 1,620,000 - 93,000 = 10,287,000 tokens/month
```

Claude Haiku 4.5のBedrock入力トークン単価（$1.10/1M tokens、日本クロスリージョン推論）で計算すると月間約$11（約1,700円）の入力トークン削減になります。

# その他の対応

そのほか以下の対応を合わせて行っています。
### 会話ウィンドウの縮小
`SlidingWindowConversationManager`の`window_size`を15→10に変更
$3~5の削減

### LTM検索結果の削減
各LTM戦略の`top_k`を引き下げ（合計18→10件）
$2~3の削減

### 定期実行エージェントの軽量化
定期的に自動でTwitter（X）に投稿させたり、ニュース収集して通知させる機能でもメインエージェントを使用していたが、代わりにメモリは共有しつつツールは最低限のものを持たせた軽量な専用エージェントを使用
$2~3の削減

## 合計削減コスト
| 施策 | 推定月間削減 |
|------|------------|
| サブエージェント分割 | $11 |
| 会話ウィンドウ縮小 | $3~5 |
| LTM検索結果の削減 | $2~3 |
| パイプライン軽量エージェント | $2~3 |
| 全施策合計 | $18~22（約2,700〜3,300円） |
