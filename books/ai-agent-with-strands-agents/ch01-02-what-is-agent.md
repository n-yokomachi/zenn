前章では10行のコードでAIエージェントを動かしました。「天気を調べて」と頼むだけで、エージェントが自らWebにアクセスし、情報を取得して回答を返してくれました。このプログラムは、普段使っているチャットボットとどこが違うのでしょうか。そして、なぜこんな動きが可能なのでしょうか。この章ではまずAIエージェントの一般的な概念を整理し、その上で前章のコードがそれをどう実装しているかを確認します。

## チャットボットとの違い

ChatGPTやClaude.aiのようなチャットボットを使ったことがある人は多いでしょう。AIエージェントとチャットボットは、どちらもLLMを使っている点では同じです。しかし、できることには明確な差があります。

| | チャットボット | AIエージェント |
|---|---|---|
| できること | 学習済み知識をもとに回答する | 外部の情報を取得し、行動した上で回答する |
| 情報の範囲 | 学習データに含まれる知識のみ | ツールを通じてリアルタイムの情報にもアクセスできる |
| 主体性 | ユーザーの質問に受動的に応答する | 目的を達成するために自ら手段を選んで行動する |

チャットボットに「東京の今の天気は？」と聞くと、「最新の天気情報は取得できません」と返されるか、学習データに含まれる過去の傾向をもとにした推測が返ってきます。一方、前章のエージェントは実際にWebサイトへアクセスし、リアルタイムの天気情報を取得して回答しました。

この違いを一言でまとめると、チャットボットは「知っていることを答える存在」、AIエージェントは「考えて行動する存在」です。

## AIエージェントの3要素

AIエージェントは、フレームワークや実装方法に関係なく、共通して以下の3つの要素で構成されています。

### LLM

LLMはエージェントの中核です。ユーザーからのリクエストを理解し、何をすべきか判断し、最終的な回答を生成します。LLMは自然言語を理解して生成する能力を持っていますが、それだけでは外部の情報を取得したり、他のシステムを操作したりはできません。

### ツール

ツールは、エージェントが外部と接点を持つための手段です。Web APIの呼び出し、ファイルの読み書き、データベースへの問い合わせなど、LLM単体ではできない操作をツールが担います。どのようなツールを与えるかによって、エージェントの能力が決まります。

### エージェントループ

エージェントループは、LLMとツールを組み合わせて自律的な動作を実現する仕組みです。一般的に以下の流れで処理が進みます。

1. ユーザーの入力をLLMに渡す
2. LLMが「何をすべきか」を判断する
3. ツールの実行が必要であれば、ツールを呼び出す
4. ツールの実行結果をLLMに戻す
5. LLMが「十分な情報が揃ったか」を判断する
6. 十分であれば回答を生成し、不十分であればステップ2に戻る

このループがあることで、開発者は「天気を調べて」という目的だけを与えれば済みます。どのURLにアクセスし、どの情報を抽出するかはエージェント自身が判断します。この自律的な判断のループこそが、チャットボットとの決定的な違いです。

エージェントループの実装方法はフレームワークによって異なりますが、概念としてはどのフレームワークにも共通しています。

## Strands Agentsにおける3要素

本書で使用するStrands Agentsは、この3要素をそのまま構造に反映したフレームワークです。

前章のコードを振り返ると、`Agent` に対してLLMのモデル（`model`）、ツール（`tools`）、行動指針（`system_prompt`）を渡すだけでエージェントが構成されていました。エージェントループはStrands Agentsが内部で処理するため、開発者が意識する必要はありません。

このように、LLM・ツール・エージェントループという一般的な概念が、Strands Agentsではそのまま直感的に対応付けられています。本書ではこの対応関係を軸に、各章で3要素を拡張していきます。

`system_prompt` はエージェントの行動指針を定義するものです。「不確かな情報は推測せず、わからないと答えてください」のように記述すれば、エージェントの振る舞いが変わります。この設計については第2章で詳しく扱います。

## まとめ

AIエージェントは、LLM、ツール、エージェントループの3要素で構成されるシステムです。これらは特定のフレームワークに依存しない一般的な概念であり、Strands Agentsはそれをシンプルな構造で表現しているフレームワークです。
