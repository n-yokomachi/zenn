## 1.1 開発環境の準備

本節では、第1章で動かすエージェントを実行できる状態までの環境構築を進めます。AWSアカウントの用意とBedrockのモデルアクセス申請、AWS CLI v2のインストール、`aws login` によるサインイン、`uv` のインストールまでを順に扱います。

### 1.1.1 前提環境の確認

対象環境はmacOS、Linux（Ubuntu）、GitHub Codespacesです。必要なものは次の3つです。

- Python 3.12以上
- AWSアカウント
- ターミナル

### 1.1.2 AWSアカウントの準備とBedrockモデルアクセス

Admin権限のIAMユーザーを事前に用意しておいてください。

次に、Amazon Bedrockのモデルアクセスを申請します。東京リージョン（ `ap-northeast-1` ）の[Bedrockコンソール](https://ap-northeast-1.console.aws.amazon.com/bedrock/home?region=ap-northeast-1#/model-catalog)を開き、モデルカタログから「Claude Haiku 4.5」を選択します。ステータスが「Available to request」であることを確認してください。

ステータスを確認したら、Anthropic FTU（First-Time Use）フォームを送信します。フォームには用途、会社情報、想定ユースケースを記入します。送信後は手動審査なしで即座に承認されます。

この申請はAWSアカウント単位で1回のみ必要です。AWS Organizations配下のアカウントには自動的に継承されます。

本書で使用するBedrockモデルIDは `jp.anthropic.claude-haiku-4-5-20251001-v1:0` です。これは日本国内ルーティングのクロスリージョン推論プロファイルです。

### 1.1.3 AWS CLI v2 のインストール

`aws login` を使うにはAWS CLI 2.32.0以上が必要です。執筆時点の最新は2.34系です。

macOSでは公式pkgインストーラーを使ってインストールします。

```bash
curl "https://awscli.amazonaws.com/AWSCLIV2.pkg" -o "AWSCLIV2.pkg"
sudo installer -pkg AWSCLIV2.pkg -target /
```

:::message
Linux / GitHub Codespacesでは、公式zipでインストールします。

```bash
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install
```
:::

### 1.1.4 aws login によるサインイン

AWS CLI 2.32.0以上では、 `aws login` コマンドでブラウザベースのサインインができます。IAM Identity Centerの事前設定は不要で、IAMユーザー、フェデレーテッドアイデンティティ、ルート認証情報のいずれもサポートしています。

サインインは次のコマンドで開始します。

```bash
aws login
```

コマンドを実行するとブラウザが開き、AWSコンソールのサインイン画面に遷移します。サインイン後、一時的なSTSクレデンシャルが発行され、 `~/.aws/sso/` 配下にキャッシュされます。アクセスキーをローカルに保存する必要はありません。

:::message
GitHub Codespacesなどブラウザを直接開けない環境では、 `--remote` オプションを使います。

```bash
aws login --remote
```

URLが表示されるので、別のブラウザでアクセスしてサインインしてください。
:::

### 1.1.5 uv によるPython環境構築

`uv` はRustで書かれたPythonパッケージマネージャーです。 `pip` や `venv` の手動管理が不要で、プロジェクトの初期化からパッケージの追加、スクリプトの実行まで一括して扱えます。公式インストーラーでインストールします。

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

インストール後の流れは、 `uv init` でプロジェクトを初期化し、 `uv add` でパッケージを追加、 `uv run` でスクリプトを実行する、という3ステップです。1.2でプロジェクトを初期化してから、このワークフローを実際に使います。

### 1.1.6 動作確認

AWS CLIのサインインと `uv` のインストールが完了したら、それぞれ確認します。

AWS認証が正しく通っているかは次のコマンドで確認します。

```bash
aws sts get-caller-identity
```

サインインに成功していれば、アカウントID、ユーザーID、ARNがJSON形式で出力されます。

`uv` のインストールは次のコマンドで確認します。

```bash
uv --version
```

バージョン番号が表示されれば、インストールは完了です。Strands Agentsのインストール確認は、1.2でプロジェクトを初期化した後に行います。

## 1.2 はじめてのエージェントを動かす

本節では、Strands Agentsを使った最小構成のエージェントを実際に動かします。「東京の今の天気を調べて教えてください」という1つの指示に対し、LLMが自律的にHTTPリクエストツールを呼び出して情報を取得し、回答を返すまでの流れを体験します。

### 1.2.1 これから作るもの

作るのはコード10数行の天気案内エージェントです。ユーザーが「東京の今の天気を調べて教えてください」と問いかけると、エージェントは自分で `http_request` ツールを使って天気情報を取得し、整形して回答を返します。ツールをいつ・どのように呼ぶかはLLMが判断します。開発者がそのロジックをコードで書く必要はありません。

### 1.2.2 プロジェクトの初期化

`uv init` でプロジェクトを作成し、必要なパッケージを追加します。

```bash
uv init my-first-agent
cd my-first-agent
uv add strands-agents strands-agents-tools
```

`uv init` で `pyproject.toml` と `.python-version` が生成され、続く `uv add` で依存関係が `pyproject.toml` に記録されます。このとき同時にプロジェクト直下へ仮想環境 `.venv/` が作成され、以降の `uv run` はこの仮想環境上で動作します。

`strands-agents` はStrands Agentsのコアパッケージ、`strands-agents-tools` はHTTPリクエストやファイル操作などを扱う組み込みツール集です。本節では `strands-agents-tools` に含まれる `http_request` ツールを使います。

### 1.2.3 エージェントの実装

エージェントの実体は `Agent` クラスの初期化と1行の呼び出しで完結します。プロジェクトディレクトリに `agent.py` を作成します。

```python:ch01/agent.py
from strands import Agent
from strands_tools import http_request

agent = Agent(
    model="jp.anthropic.claude-haiku-4-5-20251001-v1:0",
    tools=[http_request],
    system_prompt="あなたは親切なアシスタントです。日本語で回答してください。",
)

response = agent("東京の今の天気を調べて教えてください")
print(response)
```

`Agent` にモデルID、ツール、システムプロンプトを渡すだけでエージェントが構成されます。

### 1.2.4 実行

`uv run` はプロジェクトの仮想環境を自動で解決してスクリプトを実行する `uv` のサブコマンドです。`agent.py` を実行します。

```bash
uv run python agent.py
```

実行例の出力は次のとおりです。LLMが生成したテキストのため、太字やリスト形式のMarkdown記法が含まれます。実行のたびに内容は変わる場合があります。

```
東京の現在の天気を調べさせていただきます。
Tool #1: http_request
申し訳ございません。別の方法で試します。
Tool #2: http_request
東京の現在の天気情報をお知らせします：

**東京の天気情報（2026年4月16日 14:00時点）**

- **気温**: 9.9°C（約10℃）
- **天気**: ☀️ 晴れ
- **風速**: 5.4 km/h（弱い風）

現在、東京は晴れており、気温は10℃程度と少し肌寒い状況です。風も弱く、比較的穏やかな天気となっています。

お出かけの際は、気温が低めなので軽めの上着があるとよいでしょう。
```

### 1.2.5 解説

`Agent` クラスとそのコンストラクタ引数の役割を、コード上から順に整理します。

- `Agent` クラスがエージェントの本体で、LLMとの通信、ツールの管理、ループの制御をすべて担います。
- `model` にはBedrockの推論プロファイルIDを文字列で指定します。
- `tools` に渡したツールをLLMが必要に応じて呼び出します。渡したツールの数だけLLMが使える手段が増えます。
- `system_prompt` でエージェントの役割や振る舞いを指示します。
- ツール呼び出しのループや「ツールを使うべきか」の判断ロジックは自分で書いていません。

### 1.2.6 何が起きているか

エージェントが動作するとき、内部では次の流れが自動で進みます。

1. LLMがプロンプトを受け取り、ツールを使う必要があると判断します。
2. Strands AgentsがLLMにツールのスキーマ（名前・引数・説明）を提示します。
3. LLMがツール呼び出しを返し、Strands Agentsがそのツールを実行します。
4. 実行結果を再度LLMに渡し、LLMが次の判断を行います。
5. LLMがこれ以上ツールは不要と判断した時点で最終回答を生成します。

この一連の流れをStrands Agentsが自動で制御する設計を、モデル駆動アプローチと呼びます。1.3でこの概念を詳しく掘り下げます。

## 1.3 AIエージェントとは

1.2で動かしたものが何だったのかを振り返り、業界での「AIエージェント」の捉え方を整理します。

### 1.3.1 「AIエージェント」という言葉の現状

AIエージェントとは、大まかに言えば、ユーザーの代理人としてツールを自律的に選択・実行し、与えられたタスクを達成する存在を指します。1.2で動かしたエージェントもまさにこの定義に当てはまります。「東京の天気を調べて」という曖昧な指示に対し、エージェントは自分でツールの呼び出しを判断し、結果を解釈して回答を返してきました。

ただし、これはあくまで大まかな共通理解であって、業界で統一された定義は存在しません。提唱する組織ごとに強調点が異なります。本書では特定の定義を「正解」として採用せず、主要プレイヤーの定義を並べて提示します。

- Anthropic: LLMが自律的にツールをループで使うシステムをエージェントと呼びます。事前定義されたコードパスでLLMを動かす「ワークフロー」と区別し、両者を含む上位概念として「エージェントシステム（agentic systems）」という呼称も用います。
- OpenAI（Agents SDK）: ユーザーの代理として独立してタスクを達成するシステムです。LLMが意思決定し、ツールで情報収集と行動を行い、明確なガードレール内で動作します。
- Google（ADK）: コードファーストの設計思想を採り、最初からマルチエージェント構成を前提にしています。LLMエージェント・ワークフローエージェント・カスタムエージェントの3種類を組み合わせ、本番環境へのデプロイ容易性も重視します。
- AWS（Strands Agents）: モデル・ツール・プロンプトの3要素を渡せば、LLMが計画とツール呼び出しを自律的に行う「モデル駆動アプローチ」を採用しています。

共通するのは「LLMが何らかのかたちでツールを使ってタスクを進める」という骨格です。ループの強調、ガードレール、マルチエージェント、モデル駆動など、各社が重視する観点は異なります。

### 1.3.2 Strands Agentsのモデル駆動アプローチ

本書はStrands Agentsを使ってエージェントを構築するため、その設計思想を確認します。

#### モデル駆動の発想

Strands Agentsは「LLMに何をさせるか」を事前にコードで指定するのではなく、LLM自身に判断を委ねる設計を取ります。開発者が用意するのは次の3つに限られます。

- モデル: 推論を行うLLM（本書ではClaude Haiku 4.5）
- ツール: LLMが呼び出せる関数群（ `http_request` など）
- システムプロンプト: エージェントの役割や振る舞いの指示

これ以外、つまり「どのツールをいつ呼ぶか」「結果をどう解釈するか」「どこで答えを返すか」はすべてLLMが決めます。ツール呼び出しの仕組み自体は、LLMが標準で備える tool calling（関数呼び出し。function calling とも呼ばれる）機能に依存しています。

#### ワークフロー型フレームワークとの違い

LangGraphをはじめ多くのエージェントフレームワークでは、開発者がノード（処理の単位）とエッジ（ノード間の流れ）で構成されるグラフを定義し、各ステップの分岐や順序をコードで記述します。これは制御性が高い反面、フローの設計と保守が開発者の負担となります。

Strands Agentsはグラフ定義を持ちません。1.2の10数行のコードで動いたのはこのためで、フロー設計を省略できる代わりに、LLMの判断能力にエージェントの動作品質が直結します。

#### エージェントループ

Strands Agentsは内部で次のループを自動で回します。

1. 現在の会話履歴・システムプロンプト・利用可能なツール一覧をLLMに渡します
2. LLMが応答を生成します。ツールを使う必要があると判断すれば、ツール呼び出しを返します
3. Strands Agentsがツールを実行し、結果を会話履歴に追加します
4. 1に戻ります。LLMが「これ以上ツールは不要」と判断するまで繰り返します
5. LLMが最終回答を返した時点でループが終了します

会話履歴はLLMの作業記憶として機能し、ループの各反復で蓄積されていきます。1.2で動かしたエージェントも、内部ではこのループが回っていました。

なお、ループが進むほど会話履歴は増え続け、いずれはLLMのコンテキストウィンドウの上限に達します。Strands AgentsはConversationManagerと呼ばれる履歴管理の仕組みを持っており、古い会話を切り捨てる方式や要約に置き換える方式を選べます。エージェントの記憶を扱う第4章で改めて取り上げます。

#### この設計の前提

モデル駆動アプローチは、現代のLLMが「ツールを適切に選び、結果を解釈し、計画を立てる」だけの推論能力を持っていることを前提とした設計です。Claude等の高性能モデルが普及した現在、開発者がフローを細かく記述しなくてもエージェントが実用に耐える品質で動作します。

### 1.3.3 本書での扱い

本書はAIエージェントの厳密な定義を論じる本ではありません。読者がこれから作るのは「自然言語の指示を受け、外部サービスのツールを使って、ユーザーの代わりにタスクをこなすパーソナルアシスタント」です。

実装手段としてStrands Agentsを採用する理由は、モデル駆動アプローチによりフロー設計の負担がなく、章を追うごとのツール追加・記憶追加・エージェント分業といった機能拡張に集中できるためです。

第2章「エージェントの「脳」を作る」では、本章で扱った最小構成のエージェントを土台に、LLMの選定・プロンプト設計・会話の保持といったエージェントの中核を組み立てます。1.3.2で触れたConversationManagerは第4章で本格的に扱います。
