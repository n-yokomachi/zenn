本書ではStrands Agents SDKを使ってエージェントを構築しますが、同様のフレームワークはほかにもあります。主要な選択肢を紹介した上で、本書がStrands Agentsを選んだ理由を整理します。

## 主なフレームワーク

### Strands Agents SDK

AWSが公開したオープンソースのエージェントフレームワークです。`Agent` クラスにモデルとツールを渡すだけでエージェントが動く、シンプルなAPIが特徴です。Amazon BedrockやAgentCoreとの統合が前提に設計されており、モデルプロバイダーの切り替えも容易。本書のコードはすべてこのSDKをベースにしています。

### LangGraph / LangChain

エージェント開発で最も広く使われているエコシステムです。LangChainがツール連携やプロンプト管理の基盤を提供し、LangGraphがグラフベースのワークフロー定義を担います。状態管理や条件分岐など複雑な制御が得意で、コミュニティも大きい。ドキュメントやサンプルが豊富な反面、概念の多さから学習曲線はやや急です。

### OpenAI Agents SDK

OpenAIが提供するエージェントフレームワークです。Responses APIとの連携を前提に設計されており、ツール定義やハンドオフ（エージェント間の引き継ぎ）の記述がシンプル。OpenAIのモデルを中心に使う場合に最も手早く始められます。

### Google Agent Development Kit（ADK）

Googleが公開したオープンソースのフレームワークです。Vertex AIやGCPサービスとの統合が特徴で、マルチエージェント構成にも対応しています。Googleのエコシステム内でエージェントを運用したい場合に適しています。

## 比較

| 項目 | Strands Agents | LangGraph | OpenAI Agents SDK | Google ADK |
|---|---|---|---|---|
| 言語 | Python | Python / JS | Python | Python |
| クラウド統合 | AWS | 汎用 | OpenAI / Azure | GCP |
| 学習コスト | 低い | 中程度 | 低い | 中程度 |
| エコシステム | 成長中 | 大きい | 大きい | 成長中 |
| ライセンス | Apache 2.0 | MIT | MIT | Apache 2.0 |

どのフレームワークも「LLMにツールを持たせてエージェントループを回す」という基本構造は同じです。違いが出るのは、APIの抽象度、対応するクラウド基盤、そして周辺エコシステムの充実度といった点になります。

## なぜStrands Agentsを選んだのか

### コード量の少なさ

第1章のハンズオンで見たとおり、Strands Agentsではごく短いコードでエージェントが動きます。概念の数が少なく、学習コストも低めです。本書のように段階的にエージェントを拡張していくアプローチと相性がよく、章ごとに新しい要素を足していく構成が自然に書けます。

### AgentCoreとの統合

Amazon Bedrock AgentCoreを使うと、メモリ管理、ツール連携（MCP Gateway）、ランタイムのホスティングまでマネージドサービスとして利用できます。「作る」と「動かす」をひとつの技術スタックで完結させられるのは、個人開発者にとって大きな利点です。

### モデルの選択肢

Strands AgentsはAmazon Bedrockを経由して、Claude、Nova、Llama、Mistralなど複数のLLMを切り替えて使えます。特定モデルへのロックインがなく、用途やコストに応じてモデルを選べる柔軟さがあります。

## フレームワークが変わっても設計は変わらない

本書で解説するエージェントの設計思想 ── 脳（LLM）、手（ツール）、記憶、分業、自律 ── は、特定のフレームワークに依存するものではありません。Strands Agentsで学んだ設計パターンは、LangGraphやOpenAI Agents SDKなど別のフレームワークに移行しても、そのまま適用できます。フレームワークは道具であり、設計の考え方こそが持ち運べる知識です。
