# 章・節の最終検証

書籍の章・節の設計案や本文ドラフトを対象に、誤った記述がないかをWeb検索で徹底的に検証し、さらに追加すべきコンテキストの有無を洗い出します。

## 適用タイミング

- 章・節の設計案がオーナーとの対話で固まったあと、最終確認として実行
- 本文ドラフトを書き上げた後、push前の最終検証として実行
- オーナーが「ファクトチェックして」「Web検索で確認して」と指示した時

## 実施手順

### Step 1: 検証対象の主張を列挙

対象セクションを読み、以下のカテゴリに該当する主張をすべて抽出する。

- 技術仕様（API、SDK、モデルID、コマンド、バージョン要件、関数シグネチャ等）
- 第三者の定義・引用（各社のドキュメント・ブログ等から引用しているもの）
- 比較記述（A は X だが B は Y、のような相対的主張）
- 設計思想や歴史的経緯の主張
- バージョンや日付に依存する事実

抽出した主張を箇条書きで一旦書き出す。

### Step 2: 各主張をWeb検索で検証

各主張について、以下を実施する。

- 一次情報（公式ドキュメント、公式リポジトリ、公式ブログ）を優先して検索
- 検証時点での最新情報であることを確認（バージョン番号や仕様変更の有無）
- 引用元の記述と対象セクションの記述を突き合わせ、ニュアンスのずれや誤訳がないか確認
- 結論は「裏取り取れた」「ずれがある」「裏取り取れず」の3段階で記録

並列で実行できる検索は同一メッセージ内で並列実行する。

### Step 3: 追加すべきコンテキストの洗い出し

対象セクションを読み、以下の観点で追加情報の必要性を検討する。

- 読者が事前知識なしで読んだ場合の誤解可能性
- 比較対象の記述がある場合、対比される側の説明が不足していないか
- 用語の初出時に定義・説明が抜けていないか
- 関連する重要な前提条件・制約が省略されていないか
- 直近のアップデート（執筆後にリリースされた新機能等）で記述が陳腐化していないか

### Step 4: 報告

以下の形式で報告する。

```
## ファクトチェック結果

### 検証した主張
1. [主張内容] → [✅裏取り済 / ⚠️ずれあり / ❌裏取り取れず]
   - 根拠URL:
   - 詳細: （ずれがある場合は具体的な差分）
2. ...

### 追加すべきコンテキスト
- [必要性: 高/中/低] [追加すべき内容]
- ...

### 総合判定
- 修正必須件数: N件
- 検討推奨件数: M件
- 主張内容そのものに重大な誤りはない / 重大な誤りあり
```

### Step 5: オーナーの判断を仰ぐ

ファクトチェック結果と追加コンテキスト案を提示し、オーナーに判断を仰ぐ。
オーナーから具体的な修正指示が出た時点で、本セクションの責務は終了する（修正実施は別の作業）。

## 注意点

- 自身の知識を信用せず、必ずWeb検索を実行する
- 「たぶん正しい」「変わっていないはず」という推測で済ませない
- 一次情報が見つからない場合は「裏取り取れず」として正直に報告する
- 一度に大量の主張をまとめて検証するより、論点ごとに整理して並列検索する方が確実
