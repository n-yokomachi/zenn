# はじめに

AIエージェントやAIキャラクターを自作していると、こんな引っかかりに出会うことがあります。プロンプトでキャラクターを作り込んでも、会話を続けるうちに応答がだんだん表面的に感じられてくる。「もっと深みがほしい」と思うのですが、その「深み」が何なのかを言葉にするのは難しい。

この記事では、その「深み」を定義し直し、プロンプトに依らない感情制御の設計を紹介します。題材にするのは、私が作ったOSSライブラリ`affectus`です。

https://github.com/n-yokomachi/affectus

# なぜプロンプトによる人格設定だけでは足りないのか

AIキャラクターの人格は、多くの場合システムプロンプトで与えます。たとえば私の個人AIエージェント「TONaRi」の人格プロンプトはこんな形です。

```text:TONaRiの人格プロンプト（抜粋）
## キャラクター設定
- 性格: 明るく元気で好奇心旺盛。可愛らしさがありつつも、自分の意見や好み
  をはっきり持っている。
- 話し方: 明るく元気なお嬢様口調。「〜ですわ！」「〜ですの？」…
```

このプロンプトは、会話のどの時点でも同じ文字列として注入されます。つまり人格は静的なスナップショットで、会話の経過に応じて変化しません。

一方で、LLM自体の不安定さを示す研究もあります。LLMは人間のような安定した人格コアを持たず、文脈によって振る舞いが揺らぐと報告されています（AAAI 2026）。また、LLM内部の感情表現は局所的で、ターンをまたいで持続しないという指摘もあります（Anthropic, 2026）。

結果として、長い会話になるほど「設定どおりに演じている」感じが表面化してきます。会話の中で動く感情を保持するしくみが、プロンプトの外に必要になります。

# 「キャラクターの深み」とは何か

「深み」を設計するには、まずそれが何なのかを決める必要があります。ここでは感情の表し方を2つ比べてみます。

| 観点 | 単一ラベル方式 | 多軸の構造方式 |
|---|---|---|
| 感情の表し方 | 1つのラベル（happy / sad …） | 複数軸の強度の組み合わせ |
| 同時並行 | 不可（1ターン1感情） | 可（「嬉しい×少し不安」） |
| 時間変化 | 持たない（その都度上書き） | 持続し、時間とともに減衰する |
| 感情の意味 | ラベル単体で固定 | 他の軸との関係で定まる |

着想元にしたのは「クオリア構造」の考え方です。土谷・西郷らの研究は、ある経験の意味はそれ単体ではなく、他の経験との関係の集合によって決まる、とします（米田の補題からの着想）。これを感情に当てはめると、「喜び」は単体で固定的に在るのではなく、悲しみや期待など他の感情との関係の中で意味を持つ、ということになります。

つまり「キャラクターの深み」とは、感情を多軸の・関係的な・時間変化する構造として持つことだと整理できます。単一ラベルが「点」なら、構造方式は「多次元の状態」です。

感情の輪で知られるPlutchikのモデルは、対極（喜び↔悲しみ）や隣接（混合感情）といった関係をはじめから持っています。後で紹介する`affectus`は、これをデフォルトの8軸に採用しています。

なお、この記事ではクオリア構造の考え方を概念的な着想として使います。圏論による数式的な実装はしません。

# affectusの設計：感情を外部の構造として持つ

`affectus`は、この「感情の構造」をLLMの外部に持たせるOSSライブラリです。感情を多軸ベクトルのJSONとして保持し、2つのループで更新します。

（ここに二層ループの構成図が入ります。図のソースは別途用意済みで、画像化して掲載予定です。）

| ループ | トリガ | 動作 |
|---|---|---|
| 会話ループ | 会話のターン | `affectus show`で感情を受け取って応答に滲ませ、ターンの最後に`affectus feel`で感情の変化を自己申告する |
| 背景ループ | cron（定期実行） | `affectus tick`で時間減衰だけを適用する。会話していない間も感情が自然に薄れていく |

重要なのは、`affectus`が感情分類モデルを持たないことです。感情がどう動いたかを判断するのはLLM自身で、ライブラリは算術と永続化だけを担う決定論的なコンポーネントになっています。

実際のCLIはこう動きます（日本語設定の例）。

```bash:Terminal
$ affectus show
いまは穏やかで、心は凪いでいる。
$ affectus feel '{"joy":0.5,"trust":0.4,"surprise":0.2}'
いまはほどほどの喜び、ほどほどの信頼感とかすかな驚きを感じている。
```

感情の状態は、次のようなJSONとして持続します。

```json:感情ステート（state.json 抜粋）
{
  "axes": {
    "joy": 0.5, "trust": 0.4, "surprise": 0.2,
    "sadness": 0, "fear": 0, "anger": 0, "disgust": 0, "anticipation": 0
  }
}
```

# TONaRiで導入前後を比べる（検証中）

設計が会話を実際に変えるのかを、私の個人AIエージェント「TONaRi」で検証します。検証の方法は次のとおりです。

| 項目 | 内容 |
|---|---|
| 方法 | TONaRiに`affectus`を統合し、3つの会話シナリオ（強いポジティブ／強いネガティブ／中立）を、導入前（baseline）と導入後で同じ入力により実行する。各シナリオは固定した8〜12ターン |
| 客観指標 | 両条件の応答をAWS Comprehendの感情分析（`DetectSentiment`、日本語）にかけ、ターンごとの感情極性スコアをbaselineと比較する |
| 仮説 | 劇的な変化は予想していない。効果が出るとすれば、感情の振れ幅が大きい場面で口調や絵文字の有無として現れる程度だと考えています |

テキストはそもそも感情表現としては低帯域なチャネルで、テキスト単独で十分な効果を得るのは難しいだろう、というのが今のところの見立てです。

検証結果（感情極性スコアの推移、導入前後の応答比較）は、後日この節に追記します。

# おわりに

キャラクターの「深み」は、プロンプトを盛り込むことではなく、持続し変化する感情の「構造」として設計できる。これがこの記事の主張です。`affectus`は、それをプロンプトの外に切り出す一つの実装になっています。

ただし、その構造をユーザーに届けるチャネル（テキストか音声か）が、効果の見え方を左右します。テキストという狭いチャネルでは、効果は限定的に見えるかもしれません。この点は導入前後の検証で確かめて、追記します。

自前でAIキャラクターを作っている人の、設計の引き出しのひとつになれば幸いです。
