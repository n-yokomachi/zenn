### First-person（一人称）

**「私」** を一貫して使用。「僕」「筆者」「俺」の使用例はゼロ。省略（主語なし）も多用するが、明示する場合は必ず「私」。

---

### Sentence ending（文末）

**です・ます調が基本**。ただし以下のバリエーションあり：

- 体言止めによる箇条書き・見出し直後の説明文では「〜ます」を省略することがある。
- テーブルのセル内や箇条書き内は体言止め。
- 「〜です。」「〜ます。」「〜しています。」が最頻出。
- 断定的な主張は「〜です。」で締め、余韻や柔らかさを出したいときは「〜と思います。」「〜気がします。」「〜かも。」を混ぜる。
- 一文が短い・リズムを出したいところでは「〜なので。」「〜ため。」などで軽く切る。

---

### Tone（トーン）

**フランクと丁寧のブレンド**。全体はです・ます調で礼儀正しいが、砕けた相槌や自己開示が自然に挟まる。

- フォーマルな技術説明 → 軽いジョーク・自虐 → また技術説明、というリズムで場を和ませる。
- 「らしいです」「みたいでした」「なんですが」のような話し言葉的な語尾が時々登場する（特に古い記事・感想系）。
- ポモドーロタイマーのインスピレーションをゲームから得た話など、個人的な脱線エピソードをさらっと1〜2行で挟む。
- 読者への語りかけ感がある（「みんなで作って見せ合いましょう」など）。

---

### Section structure（セクション構成の慣習）

**定番の章構成パターン：**

大型技術解説記事（cadenza / agentcore 型）：
```
# はじめに
# [主題A] の中身 / 構成
## 小節
# 使い方 / 検証
## セットアップ
## 実行結果
# おわりに / おしまい
# おまけ（任意）
# 参考（任意）
```

試用・レポート記事（second-me 型）：
```
# [ツール名]とは？
# 試してみる
## インストール
## 実際に使う
# 思ったこと
# エラーと対処（任意）
```

全体像紹介記事（personal-ai-agent 型）：
```
# はじめに
# 開発方針 / 設計思想
# [機能A]
## サブ機能
# [機能B]
# 今後の予定
# おわりに
```

**見出しの深さ：** `##` までが基本。`###` は長い記事（cadenza / agentcore）でのみ使用。`####` 以深は使わない。

---

### Opening pattern（冒頭パターン）

- **「はじめに」** セクションで始めるのが主流（3記事）。最古記事（second-me）のみセクション無しで即本文。
- 冒頭で **「なぜ作ったか」「何を試したか」** を 2〜4文で簡潔に述べる。
- **リポジトリ URL** は「はじめに」の末尾または関連セクションの冒頭に `https://` の裸 URL として貼る（`@[card]` や `[text](url)` 形式は使わない場合もある）。
- 関連記事への言及は `https://zenn.dev/yokomachi/articles/...` の裸 URL で並べる（2〜3件）。
- AdventCalendar 等の告知は観察サンプル内には存在しない。

---

### Closing pattern（結尾パターン）

- **「おわりに」** または **「おしまい」** でまとめる（どちらも使う）。
- 締め方は「〜紹介しました。」「〜まとめます。」 + **1〜2文の所感・応援メッセージ** が多い。
  - 例：「パーソナルAIエージェントはいいぞ。みんなで作って見せ合いましょう。」
  - 例：「Optimizationを触ってみたい人の参考になれば幸いです。」
- 「参考になれば幸いです」の系統は agentcore 記事で使用。
- **「参考」リンク集** は `# 参考` セクションに裸 URL で列挙（agentcore 記事）、または各セクション内に分散（cadenza 記事）。
- 「おまけ」セクションで追加検証・余談をつける場合がある（agentcore 記事）。

---

### Code block conventions（コードブロック規約）

- **言語タグ＋ファイル名のセット記法** `bash: Terminal`、`json: agentcore.json（configBundlesセクション）`、`python: app/.../main.py（抜粋）`、`bash: BundleをAgentCoreに登録` など。
  - ファイル名に日本語の補足を括弧書きで添える（`（抜粋）`、`（configBundlesセクション）`）。
  - 言語タグのみ（`python:`）や言語タグなし（second-me の古い記事）も混在。
- コード前の前置き文は **1〜2文の説明** を置く。「以下は〜です」「〜はこんな形です」程度。
- コード後の反応文は **短く**（「このHookクラスは〜をベースにしています。〜だけ簡略化しています。」程度）、なくても可。
- 言語タグのない「ファイル構造」ブロック（plaintext のディレクトリ図）も使う。

---

### Image / figure conventions（画像・図の規約）

- **ホスティング：** `https://storage.googleapis.com/zenn-user-upload/...` の Zenn 標準 CDN（second-me 記事）と `/images/[記事名]/[ファイル名].png` のローカルパス（agentcore 記事）の両方が登場。
- **キャプション：** 画像の alt テキストはほぼ空（`![](...)`）。キャプションは本文中の「〜の結果がこちら。」「〜を確認できます。」など文章で代替する。
- スクリーンショット中心。Mermaid 図は不使用（使いたくないと cadenza 記事内で明示）。ASCII アートも不使用。
- プレースホルダー画像（`（スクリーンショット）`、`（構成図の画像）`）を書いたまま記事を保存することがある（personal-ai-agent 記事）。

---

### Table usage（テーブル使用頻度）

**積極的に多用**。以下の用途で安定して登場する：

- フェーズ・機能一覧（行=項目、列=役割説明）
- 比較表（行=ツール/バリアント、列=指標）
- ユーザー vs AI の主導分担
- 接触回数の仕掛け一覧

テーブルは **2〜4列構成** が多く、列幅は揃えず中身重視。ヘッダーは日本語・英語が混在しても気にしない。

### Zenn-specific notation（Zenn 固有記法）

| 記法 | 使用状況 |
|------|---------|
| `:::message` | 2〜3記事で使用。AI 使用開示、注意喚起に使う。 |
| `:::message alert` | 観察サンプルには未登場。 |
| `:::details` | cadenza 記事で output.md の全文を折り畳みに格納（大きなコンテンツの埋め込みに使う）。 |
| `@[card]` | 不使用（裸 URL が基本）。 |
| `@[tweet]` / `@[speakerdeck]` | 観察サンプルには未登場。 |

**AI 使用開示記法：** personal-ai-agent 記事では `:::message` ブロックで「この記事は人間が書き、プログラムの実装・記事の校正に生成AIを使用しています。」と冒頭に入れている。

### Emphasis decoration（強調装飾）

- **太字（`**...**`）** は適度な密度で使う。主張の核心部分・重要な概念・比較の要点をピンポイントで強調。一段落に 0〜2 か所程度。文全体を太字にすることはない。
- **emoji** の頻度は低め。`🎭`「🥺」「🤔」など感情表現・補足として文中に混ぜることがある（古い記事・感想文に多い）。最近の記事（cadenza / agentcore）では絵文字はほぼ使わない。
- **箇条書き（`-`）** は箇条書きがフィットする情報（機能一覧・注意点列挙）に使う。数が少ない（3〜5項目以下）なら箇条書きよりテーブルを選ぶ傾向。

---

### Paragraph length（段落の長さ）

- **短め〜中程度**（2〜4 文が基本）。1 文だけで段落を終わらせることも珍しくない。
- 長い技術的説明は、段落を細かく分割するよりテーブル・箇条書きに逃がす傾向。
- 段落間は空行 1 行。セクション変わりは見出しで切るのみ（追加の区切り線などは使わない）。

---

### Hedging vs. assertion（ぼかし vs. 断定）

- **断定とぼかしを意図的に使い分ける**。公式情報・自分の実験結果は断定、推測・解釈・個人的所感にはぼかしを入れる。
- ぼかし表現例：「〜らしいです」「〜と思います」「〜かも」「〜気がします」「〜と解釈できます」「〜という情報がある」。
- 情報源の信頼度が低い場合は「（公式 pricing ページからは直接抽出できず、レビュー記事を参照）」のように注記を入れる。
- 個人的な結論・主張は太字で断定する傾向（「**この用途で AI エージェントを本格活用したいなら、いずれかのツールの有料化は前提に組み込むべき**」など）。

---

### AI generation disclosure（AI 生成の開示）

- **`:::message` ブロックで冒頭に入れるパターン**が 1 記事で確認（personal-ai-agent）。
- 内容は「人間が書き、AIが実装・校正に使用した」という形の人間主導の明示。
- 全記事に必ず入れるルールではなく、選択的に入れる模様。

---

### Digressions / asides（余談・差し込み）

- **括弧書きのコメント** で補足・本音・ジョークを挟む。例：「（サンプルなので。）」「（自分）の良心に任せることにします」「（私は他に一緒にやる人がいなかったので、自分の複製が独り言を言っているのを眺めていました）」。
- **自虐・自己ツッコミ** の傾向あり：「これ以上は私のお財布に響くので検証としては中断します。」「なんか自分の複製の出来が資金力に影響されるのは微妙な気持ち🥺」。
- 個人的なエピソードは 1〜2 文で短く挿入。掘り下げない。

---

### Half-width / full-width policy（半角・全角ポリシー）

- **英数字は半角**、日本語は全角。基本的なルールは守られている。
- **英語単語と日本語の間にスペースを入れない**ことが多い（「Strands Agentsで実装し」「GitHub Copilot Freeは」など）。ただし技術文書的な文体が強い cadenza の output.md 内では半角スペースを入れている箇所あり（文脈や気分による可能性）。
- 記号まわり：`（）` は全角括弧、`()` は半角括弧、混在する。コード・コマンドは当然全角なし。
- URL は裸のまま貼る（前後にスペースや記号なし）。

## Distinctive phrases / cadences（そのまま真似る価値のある特徴的な言い回し・リズム）

- 「〇〇なのかな、と思っています。」→ 柔らかい問いかけ+所感（personal-ai-agent）
- 「〇〇はいいぞ。みんなで〇〇しましょう。」→ 締め言葉として読者を巻き込む短い呼びかけ（personal-ai-agent）
- 「というわけで〇〇した話でした。」→ 記事全体を一言でラップする締め（agentcore）
- 「〜の参考になれば幸いです。」→ 締めの定番フレーズ（agentcore）
- 「以下の〇つが〜です。」→ テーブル前の前置き文（複数記事）
- 「〜という構成です」「〜という流れです」→ フロー説明の常套句（cadenza / agentcore）
- 「なお、〜」→ 例外・補足事項を軽く断るときの前置き（複数記事）
- 「〜たところ、〇〇という結果でした。」→ 検証結果の報告フォーマット（agentcore）
- 「〜が今回の検証から得られた結論です。」→ まとめの定番締め（agentcore）
- 「（中略）」→ コードの長い部分を省略するときの表記（agentcore）

## Things to AVOID（サンプルから読み取れるアンチパターン）

- 「〜してみました！」「〜ですね！」など語尾に感嘆符を多用する（サンプル全記事でほぼゼロ）
- 「まとめ」という見出し語（「おわりに」「おしまい」を使う。「まとめ」は未使用）
- `@[card]` や `[リンクテキスト](url)` 形式での URL 埋め込み（裸 URL が基本）
- Mermaid ダイアグラムの使用（明示的に「Mermaid図表は不採用」と書いた記事あり）
- 文体のテンションが高い導入（「〜をやってみた！」系の元気すぎる冒頭）
- 長い前書き・お断り（「はじめに」は短く 3〜5 文以内）
- 記事内でモデル名・ツール名（Claude Code など）を共著者・著者として記載
- `####` 以深の見出し
- テーブルのない長い箇条書き（3 項目以上ある比較・一覧はテーブルへ）
- コード後の「どうでしょうか？」「素晴らしいですね」などの読者への問いかけ補足
