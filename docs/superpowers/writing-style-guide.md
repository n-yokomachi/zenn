# 文章生成ルール — AIエージェント構築本

本書の執筆・校正時に参照するスタイルガイド。執筆を進めながら随時更新する。

## 文体

- **ですます調**で統一する
- 一人称は使わず、主語を省略するか「筆者」を使う（多用しない）
- 読者への呼びかけは「あなた」ではなく、主語省略か「読者」を使う
- 堅すぎず、かといってくだけすぎない。実践者が同僚に共有するようなトーン

### OK例

```
Strands Agents SDKをインストールします。
実際に動かしてみましょう。
ここではタスク管理機能を題材に、ツール連携の実装方法を解説します。
```

### NG例

```
Strands Agents SDKをインストールしてください！✨
さあ、実際に動かしてみましょう！🚀
ここでは、**タスク管理機能**を題材に、ツール連携の**実装方法**を解説していきます。
```

## 禁止事項

### 絵文字

- 本文中に絵文字を使用しない
- 箇条書きの先頭に絵文字を使用しない（🔹、✅、📌 等）
- 見出しに絵文字を使用しない

### 太字（ボールド）

- 箇条書きのタイトル部分を太字にしない
- 文中で強調目的の太字を原則使用しない
- 技術用語の初出時の太字も不要（文脈で伝わる）

### AI特有の構文パターン

以下のパターンは使用しない。

| パターン | 例 | 代替 |
|---------|---|------|
| 感嘆符の多用 | 〜しましょう！ | 〜しましょう。 / 〜します。 |
| 不自然な高揚感 | さあ、始めましょう！ | では始めます。 |
| 締めの決まり文句 | いかがでしたでしょうか？ | （使わない。次のセクションに進む） |
| まとめでの内容反復 | この章では〜を学びました。次は〜 | 簡潔に1文で次章への橋渡しのみ |
| 過剰な前置き | 〜について詳しく見ていきましょう | 直接本題に入る |
| 許可・推奨の婉曲表現 | 〜してみてはいかがでしょうか | 〜してみてください |
| 無意味な接続 | それでは次に〜を見ていきましょう | （接続なしで次の見出しへ） |
| 列挙の過剰な前振り | 以下の3つのポイントがあります：| （前振りなしで直接列挙） |

### AI特有の言い回し

以下の表現は避ける。

- 「〜と言えるでしょう」→ 断定するか、根拠を示す
- 「〜することが重要です」→ 具体的に何がどう重要かを書く
- 「〜について理解を深めましょう」→ 直接説明に入る
- 「素晴らしい〜」「強力な〜」「非常に〜」→ 形容を減らし、事実を書く
- 「〜を活用する」→「〜を使う」
- 「包括的な」「網羅的な」「シームレスな」→ 具体的に言い換える
- 「〜を実現します」→「〜できます」
- 「それでは早速〜」→ 直接本題に入る

## 推奨する書き方

### 文の長さ

- 一文は短めに。体言止めも適度に使う
- 長い説明が必要な場合は、複数の短い文に分ける

### 見出し

- 簡潔にする。動詞を含めず名詞句で終わるのが基本
- 見出しだけで内容が推測できるようにする

### コードブロックの導入

- コードの前に「なぜこのコードを書くのか」を1〜2文で説明する
- コードの後に「何が起きるか」を1〜2文で説明する
- 「以下のコードを見てください」のような空虚な導入はしない

### OK例

```
エージェントの初期化にはモデルとツールの指定が必要です。

\```python:agent.py
agent = Agent(
    model="us.anthropic.claude-3-5-haiku-20241022-v1:0",
    tools=[http_request],
)
\```

これで、HTTPリクエストを実行できるエージェントが用意できました。
```

### 箇条書き

- 箇条書きは情報の列挙に使う。流れのある説明には地の文を使う
- 箇条書きの各項目は体言止めまたは用言止めで統一する
- 箇条書き項目の先頭を太字にしない

### OK例

```
このハンズオンに必要なものは3つです。

- Python 3.12以上
- AWSアカウント
- ターミナル
```

### NG例

```
このハンズオンに必要なものは以下の通りです：

- **Python 3.12以上** — 最新の機能を活用するために必要です
- **AWSアカウント** — Bedrockのモデルにアクセスするために必要です
- **ターミナル** — コマンドラインでの操作に使用します
```

### セクション間の遷移

- 無理な橋渡しはしない。見出しが切り替われば文脈も切り替わる
- 必要なら1文で次の話題を予告する程度にとどめる

### 失敗・制約の記述

- うまくいかないケースも正直に書く
- エラーメッセージは実際のものを載せる
- 「〜かもしれません」ではなく「〜の場合は」で具体的に書く

## トーンの指針

本書はブログ記事ではなく技術書である。すべての内容は検証済みの事実に基づく。

- 「試してみます」「やってみましょう」のような探索的な語り口は使わない
- 著者は内容を把握した上で解説している立場。手探り感を出さない
- 断定できることは断定する。曖昧な表現で逃げない
- ただし堅くなりすぎない。教科書ではなく、実務者が書いた実践ガイドという位置づけ
- 技術用語は文脈の中で自然に導入し、改まった定義文を挟まない
- リンクは文中に自然に埋め込む

## テクニカルライティングの原則

技術書として守るべき基本原則（3C）。

- 正確（Correct）: 数値・単位・コマンド・API名の誤記がないこと
- 明確（Clear）: 誰が読んでも同じ理解になること。複数解釈できる文を書かない
- 簡潔（Concise）: 冗長な表現を避け、一文に複数の話題を詰め込まない

## パラグラフライティング

- 一段落で扱う話題は1つに限定する
- 段落の先頭にトピックセンテンス（要点）を置く
- 「全体像を先に、詳細を後に」の順序で構成する

## 用語の統一

### 略語

- 略語の初出時は正式名を記載し、括弧内に略語を添える
  - 例: Model Context Protocol（MCP）
- 以降は略語のみで統一する

### カタカナ語

- 語尾の長音符号は原則付ける
  - 例: コンピューター、サーバー、コンテナー
  - 慣用が定着しているものはそのまま: プログラミング、コーディング
- 日本の技術コミュニティでの慣用に合わせる

### 表記揺れの防止

同じ概念には同じ語を一貫して使う。以下は本書で統一する表記の例。

| 統一表記 | 使わない表記 |
|---------|------------|
| 作成する | 作る、作製する |
| 設定する | 設定を行う |
| 確認する | 確認を行う |
| できる | することができる |
| 使う | 利用する、活用する、使用する |

（本書固有の用語統一は執筆を進めながら追記する）

## 漢字の閉じ開き

技術書で表記揺れが起きやすい語の基準。

### ひらがなで書く語

形式名詞:
- こと、とき、ところ、もの、ため、よう、わけ、はず、まま、うち、とおり

補助動詞:
- ～ていく、～ておく、～てみる、～てくる、～てほしい、～てよい、～てしまう、～てください

接続詞:
- しかし、したがって、ただし、また、ところが、あるいは、かつ、ゆえに

副詞:
- かなり、やはり、ふと、よほど

助詞:
- ～くらい、～ほど、～など

### 漢字で書く語

副詞:
- 必ず、既に、全て、特に、常に、再び、更に、例えば、初めて

## 冗長表現の簡潔化

| 冗長な表現 | 簡潔な表現 |
|-----------|----------|
| ～することができる | ～できる |
| ～を行う | ～する |
| ～を実施する | ～する |
| ～において | ～で |
| ～に関して | ～について（または省略） |
| ～という点が挙げられます | （直接述べる） |
| ～することが可能です | ～できます |

## 数字の表記

- 横書きでは算用数字を使う: 3つ、10個、第2章
- 概数は漢数字: 数十人、二十数個
- 助数詞の「か」はひらがな: 3か所、7か月
