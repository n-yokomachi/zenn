## 1.2 はじめてのエージェントを動かす

### 1.2.1 これから作るもの

- 「東京の今の天気を調べて教えて」と問いかけ、エージェントが自分でHTTPリクエストツールを使って情報を取得し、回答を返す
- コード10数行で動く最小構成
- LLMが「ツールを使うべきか」「どのツールを呼ぶか」を自ら判断する流れを体験する

### 1.2.2 プロジェクトの初期化

```bash
uv init my-first-agent
cd my-first-agent
uv add strands-agents strands-agents-tools
```

- `pyproject.toml` に依存が記録される
- `strands-agents` はコア、`strands-agents-tools` は組み込みツール集（http_request など60種類超）

### 1.2.3 エージェントの実装

ファイル: `agent.py`（リポジトリ上は `ch01/agent.py`）

```python:ch01/agent.py
from strands import Agent
from strands_tools import http_request

agent = Agent(
    model="jp.anthropic.claude-haiku-4-5-20251001-v1:0",
    tools=[http_request],
    system_prompt="あなたは親切なアシスタントです。日本語で回答してください。",
)

response = agent("東京の今の天気を調べて教えてください")
print(response)
```

### 1.2.4 実行

```bash
uv run python agent.py
```

- 実行例の出力（実際の動作ログを掲載）
- LLMが http_request を呼び、天気情報APIにアクセスし、結果を整形して返す挙動を確認
- 結果は実行のたびに変わる場合がある旨を明記

### 1.2.5 解説

- `Agent` クラスがエージェントの本体
- `model` には Bedrock の推論プロファイルIDを指定
- `tools` に渡したツールを LLM が必要に応じて呼び出す
- `system_prompt` で振る舞いを指示
- ここではツール呼び出しのループや判断ロジックを自分で書いていない点に注目

### 1.2.6 何が起きているか（簡単な解説）

- LLM がプロンプトを受け、ツールを使う必要があると判断
- Strands Agents がツールスキーマをLLMに提示
- LLM がツール呼び出しを返し、Strands Agents が実行
- 結果を再度LLMに渡し、最終回答を生成
- この一連の流れを Strands Agents が自動で回す（モデル駆動アプローチ）

### ファクトチェック済み事項（1.2）

- インポート: `from strands import Agent` / `from strands_tools import http_request`
- パッケージ: `strands-agents` と `strands-agents-tools` の2つ
- `Agent` の引数: `model`（Bedrockモデル ID 文字列を直接受け付ける）、`tools`、`system_prompt`
- 呼び出し: `agent("質問文")` のコール構文（`.invoke()` ではない）
- `http_request` ツールは `strands_tools` に存在、HTTPメソッド・URL・ヘッダー・認証等を扱える

## 1.3 AIエージェントとは

### 1.3.1 「AIエージェント」という言葉の現状

AIエージェントとは、大まかに言えば、ユーザーの代理人としてツールを自律的に選択・実行し、与えられたタスクを達成する存在を指す。1.2で動かしたエージェントもまさにこの定義に当てはまる。「東京の天気を調べて」という曖昧な指示に対し、エージェントは自分でツールの呼び出しを判断し、結果を解釈して回答を返してきた。

ただし、これはあくまで大まかな共通理解であって、業界で統一された定義は存在しない。提唱する組織ごとに強調点が異なる。本書では特定の定義を「正解」として採用せず、主要プレイヤーの定義を並べて提示する。

- Anthropic: LLMが自律的にツールをループで使うシステムをエージェントと呼ぶ。事前定義されたコードパスでLLMを動かす「ワークフロー」と区別し、両者を含む上位概念として「エージェントシステム（agentic systems）」という呼称も用いる
- OpenAI（Agents SDK）: ユーザーの代理として独立してタスクを達成するシステム。LLMが意思決定し、ツールで情報収集と行動を行い、明確なガードレール内で動作する
- Google（ADK）: コードファーストの設計思想を採り、最初からマルチエージェント構成を前提にしている。LLMエージェント・ワークフローエージェント・カスタムエージェントの3種類を組み合わせ、本番環境へのデプロイ容易性も重視する
- AWS（Strands Agents）: モデル・ツール・プロンプトの3要素を渡せば、LLMが計画とツール呼び出しを自律的に行う「モデル駆動アプローチ」

共通するのは「LLMが何らかのかたちでツールを使ってタスクを進める」という骨格だが、ループの強調、ガードレール、マルチエージェント、モデル駆動など、各社が重視する観点は異なる。

### 1.3.2 Strands Agentsのモデル駆動アプローチ

本書はStrands Agentsを使ってエージェントを構築するため、その設計思想を確認する。

#### モデル駆動の発想

Strands Agentsは「LLMに何をさせるか」を事前にコードで指定するのではなく、LLM自身に判断を委ねる設計を取る。開発者が用意するのは次の3つに限られる。

- モデル: 推論を行うLLM（本書ではClaude Haiku 4.5）
- ツール: LLMが呼び出せる関数群（http_request など）
- システムプロンプト: エージェントの役割や振る舞いの指示

これ以外、つまり「どのツールをいつ呼ぶか」「結果をどう解釈するか」「どこで答えを返すか」はすべてLLMが決める。ツール呼び出しの仕組み自体は、LLMが標準で備える tool calling（function calling）機能に依存している。

#### ワークフロー型フレームワークとの違い

LangGraphをはじめ多くのエージェントフレームワークでは、開発者がノード（処理の単位）とエッジ（ノード間の流れ）で構成されるグラフを定義し、各ステップの分岐や順序をコードで記述する。これは制御性が高い反面、フローの設計と保守が開発者の負担となる。

Strands Agentsは反対の極にあり、グラフ定義を持たない。1.2の10数行のコードで動いたのはこのためで、フロー設計を省略できる代わりに、LLMの判断能力にエージェントの動作品質が直結する。

#### エージェントループ

Strands Agentsは内部で次のループを自動で回す。

1. 現在の会話履歴・システムプロンプト・利用可能なツール一覧をLLMに渡す
2. LLMが応答を生成。ツールを使う必要があると判断すれば、ツール呼び出しを返す
3. Strands Agentsがツールを実行し、結果を会話履歴に追加
4. 1に戻る。LLMが「これ以上ツールは不要」と判断するまで繰り返す
5. LLMが最終回答を返した時点でループ終了

会話履歴はLLMの作業記憶として機能し、ループの各反復で蓄積されていく。1.2で動かしたエージェントも、内部ではこのループが回っていた。

なお、ループが進むほど会話履歴は増え続け、いずれはLLMのコンテキストウィンドウの上限に達する。Strands Agentsはこれに対処するためConversationManagerと呼ばれる履歴管理の仕組みを持っており、古い会話を切り捨てる方式や要約に置き換える方式を選べる。本書ではエージェントの記憶を扱う第4章で改めて取り上げる。

#### この設計の前提

モデル駆動アプローチは、現代のLLMが「ツールを適切に選び、結果を解釈し、計画を立てる」だけの推論能力を持っていることを前提とし、その能力を最大限活用する設計である。Claude等の高性能モデルが普及した現在、開発者がフローを細かく記述しなくてもエージェントが実用に耐える品質で動作する。

### 1.3.3 本書での扱い

本書はAIエージェントの厳密な定義を論じる本ではない。読者がこれから作るのは「自然言語の指示を受け、外部サービスのツールを使って、ユーザーの代わりにタスクをこなすパーソナルアシスタント」である。

実装手段としてStrands Agentsを採用する理由は、モデル駆動アプローチによりフロー設計の負担がなく、章を追うごとのツール追加・記憶追加・エージェント分業といった機能拡張に集中できるためである。各機能の概念と実装は2章以降で順に扱う。
